# 出国税チェッカー

日本の居住者が海外へ移住するときに問題になる「国外転出時課税制度（出国税）」に該当するかを、出国予定日から逆算して確認できるツールです。日本語・英語のバイリンガル対応。

**公開URL: https://jp-exit-tax.vercel.app**

## 背景

有価証券等を1億円以上持つ日本の居住者が海外へ移住すると、実際には1株も売っていないのに、含み益に所得税がかかります。2015年7月1日に施行された「国外転出時課税制度」です。

この制度の厄介なところは、**知らないまま出国した瞬間に、救済手段がすべて閉じる**ことです。納税猶予も、株価が下がったときの減額措置も、帰国したときの課税取消しも、どれも「国外転出の時までに納税管理人の届出をする」ことが前提になっています。飛行機に乗ってから気づいても、もう間に合いません。しかも更正・決定の期間制限は原則7年で、届出をしていれば5年に短縮されます。つまり届け出ずに出国した人は、7年間さかのぼって課税される側に立ちます。

制度の認知度は低いままです。日本語の解説記事はいくつもありますが、**自分の数字を入れて確かめられるツールは、調べた限り存在しませんでした。**

世間の誤解も多く見つかりました。とくに次の3つは、解説記事を読んだ人でも間違えます。

- **NISAは対象外だと思われている。** 逆です。売っても税金がかからない口座ですが、1億円の判定には含めます。国債や割引債のような「譲渡所得が非課税になる有価証券」も同じく含めます。非課税だから除外、ではありません。
- **暗号資産が対象になったと思われている。** 令和8年度税制改正で暗号資産は申告分離課税に移り、税法上は有価証券に近い整理をされました。改正所得税法は2026年3月31日に成立済みです。しかし国外転出時課税の対象資産には組み込まれていません。現時点でも対象外のままです。
- **「3か月前」が締切だと思われている。** 3か月前の日は締切ではなく、**納税管理人の届出をしない場合の株価の評価日**です。本当の締切は出国日そのものです。

このツールは、②「国外財産調書 提出義務チェッカー」のユーザー層（有価証券を持つ日本居住者）からの自然な導線として作りました。国外財産が5,000万円を超える人と、有価証券が1億円を超える人は、かなり重なります。

## 機能

- 6つの設問に答えると、該当・非該当・判定不能のいずれかを理由つきで返します
- **出国予定日を1つ入れるだけで、関係するすべての期限を逆算します。** 評価の基準日、納税管理人の届出期限、申告・納税の期限、担保の提供、継続適用届出書、期限延長届出書、納税猶予の納期限、帰国による課税取消しの期限。それぞれ今日から何日後か・何日経過したかを表示します
- 納税管理人を届け出るかどうかで**評価の基準日が変わる**ことを、判定に織り込みます
- 1億円のしきい値からの距離を表示し、前後1割に入る場合は概算では足りないことを伝えます
- 対象資産のチェックリスト（数えるもの9件／数えないもの4件、株式報酬をお持ちの方向けに2件）
- 納税猶予を使う場合の期限は、チェックボックス1つで足し引きできます
- 相続・贈与でも同じ課税が起きることの案内
- 含み益の概算から税額の桁を出す簡易計算（任意）
- 日英切替、URLでの条件共有、印刷・PDF出力

## 実装上の工夫

ここが本体です。迷った箇所と、その決着を書きます。

### 設問の順序は、制度の側から強制された

最初に描いたフローは「STEP1で資産、STEP3で出国」でした。1億円は最強のフィルタなので先に聞きたい、という発想です。これは成立しません。

1億円の判定は、**基準日の時価**で行います。そして基準日は、納税管理人を届け出れば「国外転出の時」、届け出なければ「国外転出の予定日から起算して3か月前の日」です。つまり出国日と納税管理人の有無が決まらないと、いつの株価で判定するのかが決まりません。**基準日が未確定のまま1億円を尋ねると、答えようのない質問になります。**

そこで順序を逆にしました。出国予定日 → 納税管理人 → 基準日が確定 → その日の時価で1億円判定。先に来る設問はどれも軽いので、離脱コストはほとんど増えていません。

この「同じ人・同じ資産でも、納税管理人を置くか否かで判定日が変わり、株価しだいで該当と非該当が入れ替わり得る」という点は、どの解説記事もさらっとしか触れていません。入力型のツールにしかできない仕事だと考え、アプリの中核に据えました。

### 「3か月前の日」を実装して、暦の穴に落ちた

国税庁のFAQに、この計算だけを扱う設問（Q6）があります。専用の設問が立っているということは、直感どおりではないということです。実際に確かめました。答えは「予定日から遡って3か月前の**応答日**」で、令和5年7月14日出国なら令和5年4月14日。「起算して」とあるので初日不算入で1日ずれるかと疑いましたが、ずれません。単純な同日3か月前でした。

**ところがFAQが答えていない穴があります。応答日が存在しない場合です。** 5月29日（平年）・5月30日・5月31日・7月31日・12月31日に出国する人には、3か月前の応答日がありません。年間5日、約1.4%。年末移住を考えれば12月31日は普通に起き得ます。

民法143条2項ただし書に倣えば「その月の末日」ですが、**国税庁は明示していません。** ここで月末に丸めてコードに埋め込み、黙っているのが一番まずいと考えました。正しそうに見える数字を、根拠なく返すことになるからです。

そこで判定不能にして、理由を出すことにしました。「出国予定日が2026.12.31のため、そのちょうど3か月前にあたる日が暦の上に存在しません（2026.09にその日はありません）。この場合にどの日を使うかは、国税庁の公表資料に書かれていません」。そのうえで回避策を書きました。日付を1日ずらすか、納税管理人を届け出れば、この問題自体が消えます。届け出れば基準日は出国日そのものになるからです。

判定できないことを判定結果として返すのは負けのように見えますが、理由が示されていれば利用者は自分で判断できます。ここは黙って丸めるより正直です。

### 「5年超」を日数で数えて、間違えた

「出国日前10年以内に国内在住期間が5年超」。最初は素直に、期間を日数で通算して1826日と比べました。そして5年線の前後90日は判定不能にしました。数え方に確信が持てなかったからです。

**この90日は、私が実装ミスを隠すために引いた線でした。**

数え方には答えがあります。ただし出国税の条文（所法60の2）に対してではなく、**瓜二つの双子の条文**に対してです。非永住者判定（所法2条1項4号）は「過去10年以内において国内に住所又は居所を有していた期間の合計が5年以下」── 文言がほぼ同一で、こちらには通達があります。

> **所基通2−4の3** 「国内に住所又は居所を有していた期間」は、暦に従って計算し、1月に満たない期間は日をもって数える。また、当該期間が複数ある場合には、これらの**年数、月数及び日数をそれぞれ合計し、日数は30日をもって1月とし、月数は12月をもって1年とする**。なお、…当該期間は、**入国の日の翌日から出国の日まで**となる。

**これは日数の合計ではありません。** 年・月・日を別々に積んでから、余った日数を30日＝1月で繰り上げます。滞在が分割されるほど、この丸めが効きます。

どれだけずれるのか、ランダム559,989ケース（滞在1〜14回・10年窓内）で突き合わせました。

| 方式 | 結論が食い違う | 5年線からの最大距離 |
|---|---|---|
| 日数の単純合計（両端算入） | 0.52% | ±16日 |
| 起算日を通達に合わせた版 | 0.23% | ±12日 |

**起算日を直しても消えません。** 日数合計方式は、どう修正しても5年線±12日で通達と割れます。最悪ケース（滞在10回）では、通達方式が「5年0ヶ月2日＝該当」、日数合計が「1814日＝5年に12日足りない＝対象外」と正反対になりました。

通達方式に置き換えました。実装が正しいことは、実務家が公開している計算例（2014.4.2起算〜2017.6.30 → 3年2ヶ月29日）を再現できることで確かめています。10年窓も、所基通2-4の2に倣って「出国日の10年前の同日から出国日の前日まで」としました。

**そして、90日のマージンは撤廃しました。**

理由は一貫性です。1億円が9,999万円のとき、このツールは「対象になりません」と判定したうえで、しきい値からの距離を出し、前後1割なら精密に計算するよう促します。判定を放棄しません。なのに5年線だけ判定不能にしていました。同じ性質のしきい値なのに、扱いが違う。

しかも判定不能にすると**タイムラインが出ません**。5年線の近くにいて1億円以上を持つ人こそ、納税管理人の届出期限を知るべき人なのに、です。

いまは「国内在住期間：5年1日／5年の線を1日超えています」と距離を出し、30日以内なら「この範囲では、住所や居所の認定、日付の記憶しだいで結論が変わります」と警告し、そのうえで判定もタイムラインも出します。断定を避けるのではなく、距離を見せて、精度がどれだけ要るかを利用者に決めてもらう。1億円で既に採っている方針に揃えました。

12月31日の出国を判定不能にしたのとは、性質が違います。あちらは**国税庁が答えを公表していない**。こちらは**双子条文の通達という答えがある**。答えがあるなら計算し、無いなら黙る。線引きはそこです。

**弱点も正直に書いておきます。** 所基通2-4の3は所法2条1項4号のものであって、60条の2のものではありません。**類推適用です。** 60条の2専用の通達は、探した限り公表されていません。他に依るべきものが無く、実務家も同じ数え方をしているので採りましたが、この点は画面にも明記しています。

同じ実装をindex.htmlとengine.jsの2箇所に持つ以上、必ず食い違います（実際、一度ドリフトしました）。ランダム60,000ケースで両者の出力が完全一致することを、テストで毎回確かめています。

### 期限には「目安」と「確定」の2種類がある

逆算タイムラインには10種類の期限が乗りますが、**確からしさが同じではありません。**

「3か月前の日」はFAQに計算例があるので確定です。「納税管理人の届出期限＝出国の時まで」も確定。「翌年3月15日（土日なら翌月曜）」も確定です。

一方、「国外転出の日から5年を経過する日」のような期限は、期間計算の解釈が資料から一つに定まりません。初日不算入で応当日の前日が満了日、という民法の原則に従えば出国日の応当日になりますが、国税庁がそう書いているわけではありません。

そこで、精度の低い項目には**「目安」のラベルを付け**、「『目安』の付いた日付は、『◯年を経過する日』の数え方が資料で一つに定まらないため、応当日で計算した概算です。実際の期限は税理士にご確認ください」と注記しました。全部を同じ顔で並べると、確定しているものまで疑わしくなります。逆に、確定していないものを確定した顔で出すのは嘘になります。

### 赤くするのは1か所だけ

最初のフロー案には「要注意（出国3ヶ月以内）：申告期限が迫っている旨を強調」という結果パターンがありました。これは制度を取り違えています。3か月前を過ぎていること自体は致命傷ではありません。納税管理人を届け出れば基準日は出国時になり、申告期限は翌年3月15日まで伸びます。

本当に取り返しがつかないのは**出国の瞬間**です。それを過ぎると、納税猶予も減額措置も帰国時の救済も、まとめて閉じます。

なので、タイムラインで色が付く行は**「納税管理人の届出はこの日まで」の1行だけ**にしました。基準日を過ぎた行はグレーになりますが、赤くはなりません。過ぎたという情報であって、警告ではないからです。テストでも「警告色が付く項目は常に1つだけ」を検証しています。

### 折りたたみの見出しは、中の設問を先取りしてはいけない

「＋ 日本国籍ではない方へ」を開くと、中に「日本国籍をお持ちですか。／はい・いいえ」がありました。**開くという行為そのものが答えなのに、開いた先で同じことをもう一度聞いていました。** 関門が二重です。

同じ構造が「＋ 届け出る予定の方へ」にもありました。見出しで「届出をしない方は開く必要はありません」と絞り込んでおきながら、中で「納税を先延ばしにする制度を使う予定はありますか。」と聞いていました。

原則として言語化すると、**見出しで絞り込んだことを、中で聞き直さない**。この原則で全部の折りたたみを点検したところ、該当はこの2つだけでした。外貨建て・株式報酬・相続・贈与は注記だけなので問題ありません。

対処は2つに分かれました。

**国籍は、設問ごと削除しました。** 調べ直すと、この設問は**判定に一切使っていませんでした**。在留資格で除外される期間は、利用者自身が期間入力から外す運用にしてあるからです。つまり注記を出すためだけのスイッチで、そのスイッチは折りたたみ自体がすでに担っていました。状態変数もURL共有のペイロードからも消しました。

**納税猶予は、チェックボックスにしました。** こちらは本物の設問です。チェックの有無でタイムラインの行数が4行から9行に変わります。ただし「はい／いいえ」の二択を折りたたみの中に隠す必要はありません。任意の追加項目は、説明ごと常に見せて、チェックボックス1つで足ります。既定は未チェック（使わない）です。

原則そのものをテストにしました。**どの `<details>` も、中に選択肢の設問（`[data-grp]`）を持たないこと。** 今後どれだけ折りたたみを足しても、同じ形の事故は起きません。

### 「まだ判定できません」は、何も言っていないのと同じ

最初の実装では、必要な項目が埋まっていないとき「まだ判定できません／必要な項目を入れてください。」と出していました。**これは何も伝えていません。** どの項目が足りないのかが分からないので、利用者は画面を上下に探すことになります。

不足項目をSTEP番号つきで列挙する形に変えました。「あと5項目に答えると判定できます」と件数を出し、□で残りを並べ、次に答えるものだけ▶と「次はここ」のタグを付けます。

ここで二度しくじりました。

ひとつめ。設問には「いいえなら以降は不要」という早期脱出があるので、素直に実装すると**残り件数が1→5と増えます。** 答えたのに増えるカウンタは、進捗ではなく不安です。まだ答えていない設問はすべて数え、**残りが単調に減る**形に直しました。

ふたつめ。STEP 2には設問が2つあります（移住の予定があるか／出国の予定日）。これをそのまま並べると「STEP 2」が2行続きます。**内部的には正しいのですが、初めて見た人には、被っているか間違えているようにしか見えません。** 画面の見出しには「STEP 2」がひとつしかないのだから、リストにも「STEP 2」はひとつしか出てはいけません。「STEP 2-1／2-2」と枝番を振る手もありましたが、それは**画面のどこにも存在しない番号**を発明することになります。両方そろって初めて消える1項目に統合しました。STEP番号がリスト内で重複しないことを、テストで固定しています。

未完了のときは結果の枠線も細くしています。最初から太い黒枠で「まだ判定できません」と宣言されると、失敗したように見えるからです。

### 未入力とエラーは違う

「海外にいた時期がある」を押すと空欄の期間入力行が現れますが、その瞬間に「このツールでは判定できません／期間の日付が正しくありません」と表示されていました。まだ入力していないだけなのに、エラー扱いです。

判定エンジンの中で、**まだ入れていない（null）**と**入れた値がおかしい（開始日が終了日より後、など）**を別の状態として区別しました。前者は未完了として不足項目に並び、後者だけがエラーになります。

### 日本語は、レイアウトが崩れても崩れて見えない

折りたたみの見出しが縦書きになっていました。「＋ 日本国籍ではない方へ」の下の説明文が、1文字ずつ縦に流れていました。

原因は単純で、`summary` を2列のgridにしているのに中の要素が3つあり（＋記号・見出し・説明文）、溢れた説明文が2行目の1列目、つまり幅1remの列に落ちていました。

**恐ろしいのは、これが英語なら一瞬で気づく崩れだということです。** 英語なら単語が枠からはみ出すか、不自然なハイフネーションが起きて「壊れている」と分かります。日本語は1文字で成立してしまうので、**幅1remの列に落ちても「縦書きにした」ように見えます。** 意図的なデザインに見えるので、レイアウトの事故だと気づきにくい。

しかもjsdomはレイアウト計算をしないので、テストでも捕まりません。対策として、CSS側で `.whofor{grid-column:2}` と2列目に固定したうえで、テストでは構造そのもの（すべての `summary` の子要素が必ず2つで、2番目が `.whofor` であること）と、CSSに `grid-column` の指定が存在することの両方を検査しています。レイアウトを計算できないなら、レイアウトが破綻しうる構造を作らせない、というアプローチです。

### 「数えるもの」と書いて、何を数えるかを書いていなかった

対象資産のチェックリストは「数えるもの」「数えないもの」という2列でした。**何を数えるのかが書いてありません。**

ここは誤解が起きる場所です。この制度は**基準日の時価**で1億円を判定します。ところが「株を1億円分持っている」と言うとき、買値のつもりで言う人は必ずいます。取得価額を入れれば、結論が変わります。含み益が大きい人ほど、対象なのに対象外と判定されます。**このツールが最も救うべき人を、取りこぼす向きの誤解です。**

見出しを「合計に入れるもの」「合計に入れないもの」に変え、それぞれの直下に何を数えるかを書きました。

> **合計に入れるもの** — 基準日の時価で数えます。買ったときの値段ではありません。
> **合計に入れないもの** — いくら持っていても、1億円の判定には関係しません。

設問と入力欄のラベルにも「時価」を入れました。「合計するといくらになりますか」→「**時価を**合計するといくらになりますか」、「合計（円）」→「**時価の**合計（円）」。

「入れない側」の注記も要ります。暗号資産を5億円持っている人が一覧を見て、載っていないから見落としたのか対象外なのか判断できない、という状態を消すためです。

そして、**見出しだけ直して中身を直しませんでした。** 折りたたみの中は「数えなくてよいものがあります」「次の2つは数えません」のまま、NISAの注記も「この判定には数えます」のまま。英語版も同じで、見出しが `Included in the total` なのに中身は `not counted` でした。語彙が2つに割れたままです。

STEP 5の資産まわりを「合計に入れる／入れない」に統一し、**STEP 5に「数える」が1つも残っていないこと**を日英両方でテストにしました。折りたたみを開いた状態で検査します。

ただしSTEP 3の「日本にいた期間が数えられません」は残します。**数えている対象が違う**からです。あちらは期間、こちらは金額。同じ言葉を別の対象に使うのは構いませんが、同じ対象を別の言葉で呼ぶのはいけません。テストにもその意図を書いてあります。

### 補助文が、存在しない選択肢を案内していた

STEP 3の補助文にこう書いてありました。「途中で数年だけ海外にいた、という場合は**「いいえ」を選ぶ**と期間を入力できます。」

ボタンは「ずっと日本」と「海外にいた時期がある」です。**「いいえ」というボタンはどこにもありません。** 英語版も同じで、`Choose no if you spent some years abroad` に対してボタンは `I spent time abroad` でした。

設問を「はい／いいえ」から具体的な文言に書き換えたときに、補助文を直し忘れたのだと思います。全STEPを機械的に点検したら、崩れていたのはこのSTEPだけでした。

実在するボタンの文言を書く形に直し、**補助文が指す選択肢が実在すること**をテストにしました。日本語は鉤括弧の中身、英語は引用符の中身と `Choose no` のような言い回しを拾い、そのSTEPのボタンの文言に含まれるかを照合します。ボタンの文言を変えて補助文を直し忘れると、機械が止めます。

### 一行に、起点の違う期間を2つ置かない

逆算タイムラインの、ある行がこう出ていました。

```
2032.08.01   あと 6年   猶予した所得税と利子税の納期限（5年4か月）
```

どちらの数字も正しいのですが、**測っている起点が違います。** 「あと6年」は今日から、「5年4か月」は出国日から。そして**どちらにも起点が書かれていません。** 読む人が矛盾と受け取るのは当然です。

全ラベルを機械的に点検したら、この2行（猶予の納期限と、延長した場合の納期限）だけが原則から外れていました。他はすでに起点を書いています ──「（**出国予定日の**ちょうど3か月前）」「（**帰国から**4か月以内に更正の請求）」。**先に書いた行が正しく、後から足した行が崩れていた**わけです。

「（出国から5年4か月）」に直し、原則そのものをテストにしました。**期間を書いた括弧には、必ず起点を書く。** ラベルを足すたびに機械が検査します。この検査が空振りでないことも、起点を消したら本当に落ちることで確かめています。

ついでに同じ性質の粗が見つかりました。継続適用届出書の行に「毎年」が3回出ていました。本文に2回、バッジに1回。**バッジが担う情報を本文で言い直さない**という検査も足しました。

### 「あと4034日」は情報ではない

逆算タイムラインは、期限までの距離を日数で出していました。「あと259日」までは意味がありますが、**「あと4034日」と言われても像が結びません。** 数字が大きいほど、伝わる量がゼロに近づきます。

1年未満は日数のまま、1年以上は年・か月に切り替えました。日付そのものは隣に出ているので、11年先の期限に日単位の精度は要りません。「あと 4034 日」が「あと 11年」になり、**文字数はむしろ減りました。** 行は増えていません。

英語版で「in 1 years 7 months」が出ました。単複です。ここでコードに `n === 1 ? 'year' : 'years'` と書くと、表示テキストがI18N辞書の外へ漏れます。辞書側に単数形のキーを持たせ、コードは「1なら単数キー」とだけ判断する形にしました。日本語には単複がないので同じ文字列が入りますが、他の言語へ広げても壊れません。

### 赤い一行が、間違っていた

タイムラインで唯一色が付く行は、こう書いてありました。

> 納税管理人の届出はこの日まで。過ぎると、納税猶予も減額措置も**帰国したときの救済も、すべて使えなくなります**

**この「帰国したときの救済も」が誤りでした。**

国税庁「あらまし」の冒頭には「納税猶予制度や各種減額措置等を受けるためには、国外転出の時までに納税管理人の届出書を提出するなどの手続が必須」とあります。私はこの一文だけを読んで、全部が消えると書きました。本文を読み切っていませんでした。

同じ資料の5ページに「**２ 納税管理人の届出をせずに国外転出する場合**」という節があり、そこにはこうあります。

> 準確定申告書の提出 → **帰国（国外転出の日から5年以内に帰国した場合に限ります。）**
> **課税取消し**：帰国の日から4か月以内に更正の請求をすることにより…国外転出時課税の適用がなかったものとして…再計算することができます。

巻末の【参考4】フローチャートも同じで、「納税管理人の届出なし」から「5年以内に帰国」へ矢印が伸び、「課税を取消し」に着地します。節構造もそれを裏付けていました。

| 節 | 内容 | 納税猶予が条件か |
|---|---|---|
| **3−1** | **帰国等による課税取消し** | **不要** |
| 3−2 | 価額下落による減額、外国税額控除 | **必要** |

**納税管理人の届出を逃して失うのは、納税猶予と、それを条件とする減額措置です。帰国による課税取消しは残ります。** 赤くする判断自体は変えていません。猶予が消えて全額即納になるのは、依然として最も重い結果だからです。ただし「すべて」は嘘でした。

いまは帰国取消しの行に「これだけは納税管理人の届出がなくても使えます」と明記しています。テストでも、赤い行に「帰国」の語が入っていないことを機械的に確かめます。

### 二重課税を、まるごと落としていた

日本は出国の時点の含み益に課税します。その後、移住先の国で実際に売れば、その国も同じ利益に課税します。**移住者にほぼ必ず起きる二重課税**を、このツールは一言も書いていませんでした。

「あらまし」3−2③に救済があります。

> 譲渡、決済又は限定相続等したときに、国外転出先の国の外国所得税と二重課税が生じる場合 → **外国税額控除を適用することができます。** 外国所得税を納付することとなる日から4か月以内に更正の請求をする必要があります。

これは**納税猶予を受けていることが条件**です。つまり「猶予を使わず出国前に納めきる」という選択は、将来の二重課税を自力で被る選択でもある。納税管理人の届出を逃すことの重さが、ここでもう一段増します。しかも**贈与・相続では外国税額控除は使えません**（「あらまし」6・7ページ）。

期限が「外国所得税を納付することとなる日から4か月」という**事象起点**なので、日付のタイムラインには載せられません。結果画面の注記として置きました。

### やらなかったこと

**為替レートを自動取得しない。** 外貨建ての有価証券は、基準日のTTM（対顧客直物電信売相場と買相場の仲値）で円に直します。TTBでもTTSでもありません（所基通60の2-8、FAQ Q10）。しかも国外転出前に申告するなら3か月前の日のTTM、後に申告するなら国外転出の時のTTM ── ここでも基準日の分岐がついて回ります。取ってきたレートは正しそうに見えて、その人が使うべきレートとは違う可能性があります。正確性を装った誤りになります。

代わりに判断基準を出しました。「合計が1億円から大きく離れているなら、おおまかな換算で足ります。1億円の前後1割に入るなら、正確なTTMで計算してください」。しきい値からの距離が分かれば、どこまで精度が要るかは利用者が決められます。

**株価も取得しない。** 同じ理由です。上場株式の基準日の終値すら自動では取りません。基準日は人によって違い、非上場株式に至っては純資産価額等を参酌した「所得税法上の時価」で、これは税理士の仕事です。

**桁を、目で数えさせない。** 金額の入力欄は、フォーカスが外れたときにカンマを付けていました。1億円は0が8つ並びます。入力中に生の数字だけを見せて0を数えさせるのは、このツールで最も危ない場所です。入力しながら3桁区切りが入る形に変えました。カーソルは「左から数字が何個目か」で復元します。位置そのもので戻すと、カンマが増えた瞬間にずれます。

**紙に入力を残す。** 印刷すると設問の欄は消えるようにしてありました。ところがこのPDFは税理士に渡す叩き台です。**何を入れたかが書いていない紙は、相談に使えません。** 印刷のときだけ現れる入力サマリーを足しました。画面には出しません。

**「わからない」という選択肢を置かない。** 最初のフロー案では時価合計を「1億円未満／1億円以上／わからない」の3択にしていました。3択にすると、しきい値からの距離が出せません。また「わからない」は逃げ道になります。証券口座の評価額合計を見れば済む話なので、選択肢からは外し、代わりに確認先を入力欄の脇に常時表示しました。

**「出国予定時期」を3択にしない。** 「3ヶ月以内／3ヶ月〜1年／1年以上先」では逆算ができません。日付を1つ入れてもらうことで、10種類の期限がすべて出ます。入力は1つ増えるどころか、選択肢を選ぶより速いくらいです。

**相続を判定しない。** 相続でも同じ課税が起きますが、準確定申告の期限は「相続があったことを知った日の翌日から4か月を経過した日の前日」で、資産を取得しない相続人も申告が必要で、海外在住の相続人が複数なら納税管理人の届出は全員の連署で、限定承認ならこの制度は適用されず別の規定でみなし譲渡課税になり、遺産分割が未確定なら法定相続分で申告してあとから修正申告か更正の請求をします。**判定に耐える設問に分解できる分量ではありません。** 案内にとどめ、税理士への導線を置きました。贈与も同様です。

**「除斥期間7年」を脅し文句にしない。** 結果画面に載せてはいますが、赤くも太くもしていません。事実として置いてあるだけです。

### 日本語まわりの実装

過去2本と同じ轍を踏まないための対策です。

日本語は文字単位で改行されるので、「所在」「国外転出」のような鉤括弧付きの用語が行末で分断されます。描画の出口に通す関数で、鉤括弧・二重鉤括弧の内側を自動で折り返し禁止にしています。個別に指定して回ると必ず漏れるので、出口で一括処理する方式にしました。テストでも「鉤括弧付きの用語がすべてnowrap指定されているか」を機械的に検査しています。

見出しに `ch` 単位の max-width をかけていません。全角1文字が約2ch分になるため、意図しない位置で折り返します。

言語切替でレイアウトが崩れないよう、要素の位置は grid で固定しています。`flex-wrap` は使っていません。テストで日英の要素数が一致することを確認しています。

「◯区分」のような数字を文言に直書きしていません。資産チェックリストの件数はデータの配列長から出しています。「1億円以上」の「以上」も、RULESの比較演算子（`gte`）から引いています。

## 技術

単一のHTMLファイルです。フレームワークもビルドツールも使っていません。外部リソースの読み込みもゼロで、CDNもWebフォントも使っていないので、`file://` で開いてもiframeの中でも動きます。localStorageとsessionStorageも使っていません。

制度データはすべて `RULES` オブジェクトに集約し、ロジックから分離しています。しきい値、居住期間の要件、基準日の計算規則、対象資産の分類、10種類の期限規則、税率、出典URL。改正があってもここだけを差し替えれば済みます。

表示テキストは `I18N` 辞書に完全集約しています。日英で180キー、著作権表記も含めて直書きはありません。起動時に自走するチェックがあり、日英のキー数の不一致と、英語辞書に日本語が残っていないかをコンソールに出します。このチェックは何があっても本体を止めないようにしてあります（下記）。

入力条件はハッシュ（`#s=`）にBase64で載せて共有できます。ペイロードにバージョンを持たせ、未知のバージョンでも読める鍵だけを拾うようにしてあります。壊れたハッシュを渡されても落ちません。

印刷・PDF出力を標準装備し、`@media print` で色を保持します。

### 検証

数値とロジックは、実際に動かして確かめています。

- 判定エンジンの単体テスト 80件
- **出荷するindex.htmlからコードを取り出して実行**するテスト 121件。テスト用に書いたコードが通っても、出荷物が通らなければ意味がないためです。2026年から2029年の全出国日パターン522件について、検証済みエンジンとの一致も確認しています
- jsdomで実際に画面を操作するテスト 203件。フォーカスが飛ばないこと、折りたたみが閉じないこと、言語切替で要素数が変わらないこと、警告色が1つだけであること、壊れたハッシュで落ちないこと
- **機能の欠けた環境を再現するテスト 57件**（後述）

計461件。境界値は1億円ちょうど、5年ちょうど、12月31日出国、閏年の5月29日、3月15日が土日になる年（2026年から2040年まで総当り）を含みます。

この過程で、**テストの側が7回間違っていました。** 符号を逆にしていたもの、自分のCSSコメントに正規表現が反応していたもの、「経過済みの期限があるはずだ」と決めつけていたもの、`navigator.language` のスタブを忘れて英語版を掴んでいたもの、出国予定日を取り違えて境界の日付を間違えたもの、経過日数を実日付に依存した固定値で書いて日替わりで壊れたもの。3つ目は、納税管理人を届け出て出国96日前なら全期限が未来なので、経過済みの行が存在しないのが正しい挙動でした。実行しなければ、どれも気づきませんでした。

### 補助機能が本体を殺した

260件のテストを通したあと、プレビュー環境で真っ白になりました。`console.info is not a function`。

I18N辞書の日英キー数が一致しているか、英語辞書に日本語が残っていないかを起動時に自走チェックする関数を仕込んでいたのですが、その結果を `console.info` で出していました。サンドボックス内のiframeのように `console` が痩せている環境では、ここで例外が飛んで**本体の描画ごと止まります。**

テストが素通りしたのは、jsdomが `console.info` を持っていたからです。**テスト環境がたまたま恵まれていたせいで、劣悪な環境で落ちるコードが通ってしまいました。**

開発用の補助機能が本番の描画を止めてよい理由はありません。自走チェック全体を try/catch で囲みました。

ここで一度しくじっています。フォールバック先を `console.warn` にしたところ、**成功メッセージが警告として黄色く表示されました。** 通っているのに警告を出すのは、ただのノイズです。オオカミ少年になって、本当の警告が埋もれます。**通っているときは黙り、問題があるときだけ喋る**形に直しました。無音であることと、辞書をわざと壊せばちゃんと鳴ることの両方をテストに固定してあります。無音化しすぎて検知力を殺しては本末転倒だからです。

ついでに同じ性質の危険を洗いました。`history.replaceState` はサンドボックス内で SecurityError を投げます。`location.origin` は blob: や srcdoc では `"null"` という文字列になります。`navigator.clipboard` は http:// では存在しません。どれも共有ボタンとリセットボタンで踏みます。すべて握りつぶして、機能を落としても本体は生かす形にしました。

そのうえで、**壊れた環境を再現する回帰テストを足しました。** console が無い / console が例外を投げる / history が封じられている / clipboard が無い / clipboard が拒否する / file:// で開く。今回のバグそのものを最初のケースとして固定してあります。同じ穴に二度落ちないためです。

同じ発想で、**日付も偽装します。** このツールは「今日」を実時刻から取るので、テストが実日付に依存すると日替わりで壊れます。実際、経過日数を「76日」と固定値で書いていたテストが、日付が変わった翌日に落ちました。値を計算して比べる形に直したうえで、JSDOMの `Date` を差し替えて、年末・年始・閏日・3月15日が日曜になる年など、日付まわりが荒れる6日について毎回検査しています。

## In English

Japan taxes unrealised capital gains when a long-term resident emigrates holding 100 million yen or more in securities. You do not have to sell anything; departure itself is the taxable event. The rule has been in force since July 2015 and remains poorly known.

Three things make it worth a tool rather than an article.

**The valuation date is not fixed until you decide something most people have never heard of.** If you register a tax agent with the tax office before leaving, your securities are valued at the date of departure and you file by 15 March of the following year. If you do not, they are valued exactly three months before your planned departure, and you must file before you board. Same person, same portfolio, two different valuation dates — and with a volatile position, two different answers to the 100 million yen question. This forces the order of the questions: the date and the tax agent must come before the money.

**Registration is the only door, and it closes at the moment of departure.** Deferral, the reduction when prices fall, and the cancellation if you come home within five years all require the registration to have been filed before you leave. Miss it and every remedy is gone at once. The assessment period is seven years by default, five if you registered. So the timeline in this tool colours exactly one line red, and it is not the three-month mark that everyone assumes.

**The calendar has a hole the tax authority has not filled.** "Exactly three months before" means the anniversary day — the NTA's FAQ works the example: leave on 14 July, value on 14 April. But five days a year have no such anniversary: 29, 30 and 31 May, 31 July, and 31 December. Civil Code reasoning suggests the last day of the month; the NTA does not say. Rounding silently would produce a number that looks right and is not, so the tool refuses to decide, explains why, and points out that registering a tax agent makes the problem vanish. The same restraint applies within ninety days of the five-year residence line.

The tool does not fetch exchange rates or share prices. A fetched rate looks authoritative and is probably the wrong one for that taxpayer's valuation date. Instead it shows the distance from the threshold, which is what actually determines how much precision you need: far away, estimate freely; within ten per cent, go and get the real closing price.

Built as one HTML file with no framework, no build step and no external requests. All statutory data lives in a single `RULES` object; all display text lives in a 180-key bilingual dictionary. 461 tests run against the shipped file itself, including a jsdom pass that drives the actual interface and a suite that boots the page into deliberately impoverished environments — no `console`, a `console` that throws, `history` blocked by a sandbox, no clipboard, opened from `file://`. That last suite exists because a startup self-check logged through `console.info` and took the whole page down in a sandboxed preview, after 260 tests had passed. The tests had simply been running somewhere too comfortable.

**This tool is for residents of Japan.** The rule described is Japanese law and has no application elsewhere. The English interface exists so that English speakers can see what the tool does; it omits statutory references and procedural detail deliberately.
